February 15, 2026

Email Verification

Added email verification to the registration flow today. Up until now, anyone could register with any email address and immediately get access — fine for development, but definitely not something I'd want going to production. Now when you register, you get a 6-digit code sent to your email and you have to enter it before your account activates.


On the backend, I added an EmailVerification model that stores the code with a 15-minute expiry and a max of 5 attempts. Registration now creates users as inactive (is_active=False), and the verify endpoint activates them and returns tokens. There's also a resend endpoint with a 60-second cooldown so people can't spam it. If someone tries to log in before verifying, they get a specific error that tells the iOS app to redirect them to the verification screen instead of just showing a generic failure.


The iOS side was where the interesting bugs showed up. The first version looked like it worked, but when I actually tested it on the simulator, registering just kicked me back to the login screen. Turned out there were three interrelated issues: ContentView was destroying the registration NavigationStack when isLoading toggled (because it used that flag for the splash screen), DRF's ValidationError was wrapping the login guard's response values in arrays that iOS couldn't decode, and DRF field errors like {"email": ["already exists"]} weren't being parsed at all. Fixed all three — separated isCheckingAuth from isLoading, moved the login guard to return a proper Response() instead of raising ValidationError, and added a fallback parser for DRF's field error format.


575 backend tests (38 new) and 116 iOS tests (8 new), all passing. Staff invites still bypass verification since the invite email itself is the proof of ownership.

February 8, 2026

React Admin SPA

Finally got to the frontend. After nine steps of backend-only work, today I built the React admin SPA from scratch. Scaffolded a Vite + React 19 + TypeScript project at backend/admin_frontend/, wired up Tailwind v4 with all the design tokens from STYLES.md, and set up ESLint 9 and Prettier per the code standards doc.


The app has four pages: login, dashboard, audiobook list, and import. The login page checks that you're a staff user before letting you in. Dashboard shows stat cards (total audiobooks, chapters, segments, tagging pipeline breakdown) plus a recent imports table — I added a new backend endpoint for that so the frontend doesn't have to fetch everything and count it itself. The audiobook list has search and pagination, and the import page lets you search LibriVox and import books with one click.


For auth I built a JWT interceptor with a refresh token mutex — if your access token expires mid-request, it refreshes once and replays any queued requests. If the refresh fails, you get kicked to login. The whole thing proxies through Vite's dev server so there's no CORS hassle.


Had to install Node.js first since it wasn't on the machine yet (quick brew install node). Also learned that eslint-plugin-react-hooks v7 has a new set-state-in-effect rule that's pretty strict — had to restructure the AuthContext to initialize loading state from localStorage synchronously instead of setting it in the effect body.


27 source files, 201 backend tests passing (5 new for the dashboard stats endpoint), and all frontend lint/format/typecheck clean. Next up is Step 11: the tagging tool with wavesurfer.js.

February 7, 2026

Backend Foundation + Pre-Screening Services

Cranked through all the backend foundation steps today — models, auth, catalog, LibriVox import, and filter API. The entire backend is now in place.


Started with the database models. Built out Author, Genre, Audiobook, and Chapter in the catalog app, plus FilterCategory and FilterSegment in the filtering app. Used DecimalField for segment timestamps to avoid float rounding issues during playback. Wrote a seed migration for all 7 filter categories and got everything registered in Django admin. Then built the auth API (register, login, token refresh, profile, password change) and the catalog API (audiobook list with search/filtering, detail with chapters, stream URLs).


Step 5 was the LibriVox import service — search their API, import a full audiobook with authors, genres, and chapters, all idempotent. Caught a few quirks where the real LibriVox API didn't match the reference docs I'd written. Imported books default to unpublished so an admin has to review them first.


Wrapped up with Step 6 — the Filter API. Two sets of endpoints: iOS-facing ones that return filter categories and segments grouped by chapter (only manual and ai_confirmed, with ?category= and ?severity= filters), and admin endpoints for full segment CRUD plus bulk create. The severity filter is "at or above" so ?severity=moderate returns moderate and severe. Bulk create returns 201 or 207 multi-status depending on whether all segments pass validation.


With the backend foundation locked in at 140 tests, I kept going and started the LLM pre-screening pipeline — the system that automatically identifies objectionable content so admins don't have to tag everything from scratch.


Built the TranscriptionService first. It wraps faster-whisper to transcribe chapter audio into text with word-level timestamps. Lazy-loads the Whisper model so it's not eating memory at import time, handles downloading audio to a temp file, runs transcription, and tracks status through PENDING to PROCESSING to COMPLETED/FAILED. It's idempotent — if a chapter already has a completed transcript it just returns it. Failed transcripts can be retried too, which meant clearing the old error message on retry (caught that one in tests).


Then built the ContentAnalysisService, which runs a two-stage analysis. Stage 1 is a deterministic word-list scan — match each word against a JSON file of profanity, slurs, and blasphemy. Fast and reliable. Stage 2 sends the transcript to Ollama in 30-second chunks with 5-second overlap, asking the LLM to identify violence, sexual content, crude humor, and other stuff a word list can't catch. The LLM returns structured JSON with category, severity, and word indices that get mapped back to timestamps. If Ollama is down or returns garbage JSON, it logs a warning and moves on — no crashes. Both services create FilterSegment records with source='ai_suggested' so admins can review and confirm them in the tagging tool later.


171 tests passing now (31 new, all mocked — no actual Whisper or Ollama calls). Lint clean. Next up is Step 9: pipeline orchestration to wire the services together with admin API endpoints and a management command.

January 27, 2026

Back at it - Full Architecture Plan Complete

It's been a while. School and life got in the way, but I'm finally back on Audio Nexus and I came back with a completely different mindset. Instead of just jumping into code like I did before, I sat down and planned the entire system top to bottom. I went back through every single prototype repo I've built over the last year and a half — the Tkinter desktop app, the FastAPI server, the Kivy admin, the UIKit prototype, the SwiftUI player, all of it. I pulled out what worked, what didn't, and what patterns I want to carry forward.


The biggest shift is that I'm going fully self-hosted. I spent so much time last year fighting with Spotify's API, getting rejected by OverDrive, Audible, and Hoopla, trying to find some platform that would let me control playback at the segment level. None of them do. And Spotify's ToS blocked the auto-profanity detection I built with OpenAI. So I'm done trying to make third-party platforms work. For the MVP I'm using LibriVox public domain audiobooks — they're free, have an API, and the audio files are hosted on Archive.org permanently. That gives me everything I need to prove the concept without licensing headaches.


The tech stack is completely different from where I started. Backend is Django with Django REST Framework — I picked it for the admin panel, the django-organizations library for households later, and the strong audio processing ecosystem. The admin frontend is a React SPA with wavesurfer.js for the waveform tagging tool. And the iOS app is SwiftUI with AVFoundation, which I'm excited about because the Audio-Player prototype I built already proved that AVPlayer works for what I need.


I designed the full database schema, every API endpoint, the iOS screen-by-screen navigation, the React admin component structure, infrastructure and deployment, LibriVox integration details, the audio pipeline, DRM strategy, and legal considerations. The plan has 12 detailed sections covering everything. I even designed a rip-out strategy for LibriVox so when I eventually move to publisher content, I just delete one adapter class and a couple admin endpoints and everything else keeps working.


One of the things I'm most excited about is the LLM pre-screening pipeline. Remember how the old Tkinter app used the OpenAI Whisper API to transcribe audio and scan for profanity? I'm doing the same thing but fully local now. faster-whisper for speech-to-text with word-level timestamps, and then Ollama running a local LLM for deeper content analysis — violence, sexual content, crude humor, stuff the word list can't catch. The AI creates suggested filter segments and then a human admin reviews and confirms them in the tagging tool. No cloud APIs, no ToS issues, no file size limits.


I also restructured the project into separate repos — one for the backend, one for the iOS app, and this dev log moved into the parent repo as its own thing. Each repo has independent git history and CI/CD. The parent repo just holds the project plan and documentation.


I'm honestly feeling really good about where things are. The plan is solid, it accounts for the lessons from every prototype I've built, and it's scoped properly — Phase 1 is just playback with real-time segment skipping, the tagging tool, and LibriVox content. No payments, no households, no store. Just the core feature. I'm ready to start writing code.

What is Audio Nexus?

Audio Nexus is an audiobook filtering software. The idea was inspired by services like VidAngel, Enjoy, and ClearPlay that allow users to enjoy their favorite movies and TV shows without all the swearing or explicit content. I love to read and wanted to create a service that did the same for audiobooks. So Audio Nexus was born. This project will allow users to listen to audiobooks and filter out explicit language and other explicit content.

What is this site?

This site is essentially a blog about Audio Nexus. I will use it to post updates about the project, issues I'm running into, and give insights into my though process as the project develops and takes shape.

How is the project going?

Audio Nexus is taking shape nicely. I have been programming the project using python. I created a basic media player that allows me to load in the audio file. I then send the file to OpenAI with the API to create timestamps/transcript for the audio. From there I process the timestamps to create a language filter automatically. But I'm realizing that I will probably need to create an easy way for human review. Not only to tag missed explicit language, but more importantly to tag other explicit content. I don't want to rely on an algorithms to try and create a filter for sexual or crude content. They are so subjective and don't have a clear cut start and end. So like, VidAngel I'm going to have to use human review and intuition to mark unsafe content.


So far the project is working okay with python. So I may still build the end user software with python. But right now using python to create the tags is pretty messy. I've been doing some research on which programming language is best when working with audio and the general consensus is C++. So I think I'm going to try and build the Admin side of the project with C++. But I've never worked with C++ so it'll be an interesting transition and learning process.
